[4] 'Normal': ________________________________________________________________________________
[5] 'Normal': Zeptai es una plataforma SaaS multi-agente de IA Generativa para PYMEs desplegada en Docker. El stack tecnologico es: Python + Flask (backend), PostgreSQL + pgvector (base de datos vectorial), OpenAI API (GPT-4o, Whisper, DALL-E 3, embeddings), RunwayML (video), y una PWA en Jinja2 + Tailwind CSS.
[6] 'Normal': La arquitectura se organiza en 4 capas:
[7] 'Normal': • Capa de entrada: app.py inicializa Flask, registra blueprints y migra la BD al arrancar
[8] 'Normal': • Capa de datos: core/db_models.py define 14 modelos SQLAlchemy sobre PostgreSQL
[9] 'Normal': • Capa de rutas: routes/ con 3 blueprints: web (HTML), api (JSON), knowledge (RAG)
[10] 'Normal': • Capa de logica: modules/ contiene agentes, servicios y orquestadores
[11] 'Normal': El despliegue usa Docker Compose con 4 contenedores independientes:
[12] 'Normal': • zeptai_app: aplicacion Flask principal, puerto 5000
[13] 'Normal': • zeptai_mcp: servidor MCP con herramientas externas, puerto 8001
[14] 'Normal': • zeptai_scheduler: APScheduler para tareas proactivas diarias
[15] 'Normal': • zeptai_db: PostgreSQL 15 con extension pgvector para embeddings
[16] 'Normal': 1. app.py — Punto de entrada
[17] 'Normal': ________________________________________________________________________________
[18] 'Normal': Es el fichero que Python ejecuta al arrancar. Hace cuatro cosas en orden: carga variables de entorno, inicializa Flask con su configuracion, prepara la base de datos y registra los tres blueprints de rutas.

[28] 'Normal': with app.app_context():
    db.create_all()

[31] 'Normal': conn.execute(text("CREATE EXTENSION IF NOT EXISTS vector"))

[50] 'Normal': BusinessProfile
[51] 'Normal': Es el modelo central. Cada usuario del sistema tiene exactamente un BusinessProfile. user_phone es la clave primaria y tambien la clave foranea que relaciona todos los demas modelos. Campos destacados:
[52] 'Normal': • password_hash: contrasena almacenada como hash PBKDF2-SHA256 con salt aleatorio (Werkzeug). Nunca se guarda la contrasena en texto plano
[53] 'Normal': • features: JSON con flags de funcionalidades activadas por plan (bot_enabled, etc.)
[54] 'Normal': • system_prompt: texto personalizado que define como se comporta el agente con los clientes de ese negocio

[56] 'Normal': • active_agents: lista JSON con los IDs de agentes proactivos activados por el usuario
[57] 'Normal': • agent_config: JSON anidado con la configuracion especifica de cada agente (politicas de devolucion, etc.)
[58] 'Normal': • push_subscription: JSON con los datos de suscripcion a Web Push para notificaciones PWA
[59] 'Normal': Ticket
[60] 'Normal': Representa un recibo o factura procesada. Cuando el usuario sube una foto de un ticket, GPT-4o Vision extrae los datos y se crea un registro Ticket en la BD.
[61] 'Normal': • image_path: ruta relativa al fichero de imagen guardado en static/uploads/
[62] 'Normal': • status: pending (recien subido) o processed (datos extraidos correctamente)
[63] 'Normal': • concept, total, nif, provider, date: datos fiscales extraidos por Vision API
[64] 'Normal': ChatMessage
[65] 'Normal': Almacena cada turno de la conversacion. El historial se recupera en cada nueva peticion para dar contexto al agente. Campos clave:
[66] 'Normal': • role: user, assistant o tool. OpenAI exige este formato para el historial de mensajes
[67] 'Normal': • tool_call_id + name: necesarios para los mensajes de resultado de herramienta en el protocolo de tool calling
[68] 'Normal': Limite automatico: si hay mas de 50 mensajes, el sistema borra los mas antiguos hasta quedarse con 40. Esto evita que el contexto crezca indefinidamente y encarezca las llamadas a GPT.
[69] 'Normal': KnowledgeChunk
[70] 'Normal': Es el corazon del sistema RAG. Cada chunk es un fragmento de texto del negocio con su embedding vectorial de 1536 dimensiones.
[71] 'Normal': • source_type: wizard (configuracion del negocio) o document (PDF subido por el usuario)
[72] 'Normal': • source_name: nombre del documento de origen
[73] 'Normal': • content: texto del fragmento (max 1600 caracteres)
[74] 'Normal': • embedding: vector de 1536 floats en pgvector. En SQLite se guarda como texto serializado como fallback
[75] 'Normal': LLMCall
[76] 'Normal': Registra cada llamada a un modelo de IA con sus metricas. Permite calcular el coste real de la plataforma y detectar ineficiencias.
[77] 'Normal': • model: gpt-4o, gpt-4o-mini, whisper-1, dall-e-3, etc.
[78] 'Normal': • stage: en que parte del codigo se hizo la llamada (council_opinion, rag_embed, etc.)
[79] 'Normal': • prompt_tokens + completion_tokens: tokens consumidos. El coste se calcula como tokens * precio_por_modelo
[80] 'Normal': • latency_ms: tiempo de respuesta en milisegundos
[81] 'Normal': Otros modelos
[82] 'Normal': • Appointment: citas: fecha, hora, cliente, telefono. end_time se anadi o con ALTER TABLE en migracion automatica
[83] 'Normal': • Grant: subvenciones disponibles con sector, importe, deadline y lista de usuarios ya notificados
[84] 'Normal': • SynergyMatch: matches B2B entre dos usuarios con score (0-100) y razon generada por IA
[85] 'Normal': • Notification: alertas in-app: titulo, mensaje, tipo, leida/no leida
[86] 'Normal': • GeneratedDocument: documentos generados: propuestas PDF, imagenes PNG, slides PPTX, videos MP4
[87] 'Normal': • ActivityLog: auditoria de acciones del sistema con metodo estatico .log() para registrar facilmente
[88] 'Normal': • ChatFeedback: valoraciones +1/-1 del chatbot por mensaje
[89] 'Normal': • RagRetrieval: log de consultas RAG: query, chunks devueltos, score promedio

[144] 'Normal': 4. modules/agents/ — Motor del agente principal
[145] 'Normal': ________________________________________________________________________________
[146] 'Normal': 4.1 manager.py — AgentExecutor
[147] 'Normal': Es el cerebro de la plataforma. Cuando el usuario envia un mensaje, AgentExecutor.execute() orquesta todo el proceso.
[148] 'Normal': El flujo completo de execute()
[149] 'Normal': • Paso 1: guardar el mensaje del usuario en ChatMessage (BD)
[150] 'Normal': • Paso 2: _build_rag_system_prompt(): buscar los 5 chunks mas similares a la query del usuario en pgvector y anadirlos al system_prompt. Esto es RAG en accion: el agente sabe sobre el negocio concreto del usuario
[151] 'Normal': • Paso 3: llamar a GPT-4o con el historial de mensajes + system_prompt enriquecido + TOOLS_SCHEMA (tool_choice=auto)
[152] 'Normal': • Paso 4: si GPT decide usar una herramienta (tool_call), ejecutarla con _process_tool_calls()
[153] 'Normal': • Paso 5: si hubo tool calls, hacer una segunda llamada a GPT con los resultados de las herramientas para que genere la respuesta final en lenguaje natural
[154] 'Normal': • Paso 6: guardar la respuesta del asistente en ChatMessage y devolver al frontend
[155] 'Normal': Por que dos llamadas a GPT?
[156] 'Normal': El protocolo de function calling de OpenAI funciona asi: en la primera llamada, GPT decide que herramienta usar y con que parametros pero NO genera la respuesta final. Hay que ejecutar la herramienta por nuestra cuenta y hacer una segunda llamada pasando el resultado para que GPT genere la respuesta en lenguaje natural. Esto da control total sobre la ejecucion de las herramientas.
[157] 'Normal': _build_rag_system_prompt()
[158] 'Normal': Combina el system_prompt configurado por el usuario con los chunks RAG mas relevantes. Llama a retrieve_chunks(user_phone, query, top_k=5) que hace una busqueda de similitud coseno en pgvector: calcula la distancia entre el embedding de la query y todos los embeddings guardados, y devuelve los 5 mas cercanos.
[159] 'Normal': SELECT content FROM knowledge_chunk
ORDER BY embedding <=> query_embedding
LIMIT 5
[160] 'Normal': El operador <=> es la distancia coseno de pgvector. Mas cercano = mas semanticamente similar.

[174] 'Normal': 5. modules/services/ — Servicios de soporte
[175] 'Normal': ________________________________________________________________________________
[176] 'Normal': 5.1 embeddings.py — Pipeline RAG
[177] 'Normal': Es el servicio mas importante del sistema. Gestiona todo el ciclo del RAG: ingestión de documentos, generacion de embeddings y recuperacion por similitud.
[178] 'Normal': Parametros de chunking
[179] 'Normal': CHUNK_SIZE    = 1600  # caracteres por fragmento
[180] 'Normal': CHUNK_OVERLAP = 400   # solapamiento entre fragmentos consecutivos
[181] 'Normal': El solapamiento de 400 caracteres evita perder informacion en los bordes de los chunks. Si una frase importante esta entre dos chunks, el solapamiento garantiza que aparece completa en uno de ellos.
[182] 'Normal': ingest_wizard_chunks()
[183] 'Normal': Cuando el usuario configura el wizard, cada campo (sector, servicios, horarios, FAQ, politica de devoluciones) se convierte en un chunk separado. Para cada chunk: 1) genera el embedding con text-embedding-3-small, 2) crea un KnowledgeChunk con source_type=wizard, 3) guarda en BD.
[184] 'Normal': ingest_document()
[185] 'Normal': Para PDFs: extrae el texto con PyPDF2 o pdfplumber. Para TXT/MD: lee directamente. Luego trocea el texto en chunks de 1600 caracteres con 400 de solapamiento, embebe cada uno y guarda en KnowledgeChunk con source_type=document.
[186] 'Normal': retrieve_chunks()
[187] 'Normal': Recibe la query del usuario, genera su embedding, y ejecuta la busqueda de similitud coseno en pgvector. Devuelve los top_k chunks mas relevantes con su score de similitud. Estos chunks se inyectan en el system_prompt del agente.

[203] 'Normal': 6.3 business_health.py — Coach financiero
[204] 'Normal': _get_financial_stats() calcula el gasto del mes actual, el del mes anterior y la proyeccion al final de mes (extrapolacion lineal). _generate_insight() llama a GPT con esas cifras y genera un comentario motivacional y accionable en menos de 2 frases. Se envia como notificacion in-app cada manana.
[205] 'Normal': 6.4 networker.py — Matchmaking B2B
[206] 'Normal': run_daily_networking() analiza el perfil de gastos del usuario actual (proveedores frecuentes, sectores) y lo compara con otros usuarios de la plataforma. _analyze_synergy_deep() usa GPT-4o para evaluar si hay sinergia real: por ejemplo, un gimnasio que compra mucho en tiendas de nutricion puede tener sinergia con una tienda de suplementos. Si el score supera 80/100, crea un SynergyMatch y notifica a ambas partes.
[207] 'Normal': 6.5 post_sales.py — Postventa configurable
[208] 'Normal': handle_inquiry() gestiona consultas de clientes sobre pedidos, devoluciones, cambios y garantias. Lee la configuracion especifica del usuario (agent_config.post_sales.*) para aplicar sus politicas concretas. Clasifica la consulta con GPT (RETURN/EXCHANGE/STATUS/WARRANTY/COMPLAINT) y genera una respuesta respetando las politicas configuradas.
[209] 'Normal': 6.6 admin_redactor.py — Procesamiento de imagenes
[210] 'Normal': Cuando el usuario sube una imagen, classify_image_intent() usa GPT-4o Vision para decidir si es un recibo (->extrae datos fiscales) o un borrador/apunte (->genera presupuesto PDF). process_image_request() orquesta el flujo: Vision API, extraccion de datos, generacion PDF con FPDF, guardado en disco.
[211] 'Normal': 6.7 marketing_agent.py — Generacion de contenido
[212] 'Normal': generate_marketing_content() enruta segun el tipo solicitado:

[218] 'Normal': 7. modules/council/ — Consejo Pyme
[219] 'Normal': ________________________________________________________________________________
[220] 'Normal': 7.1 orchestrator.py — CouncilManager
[221] 'Normal': Implementa el debate entre tres asesores virtuales. run_session() es un generador asincrono que yield eventos SSE conforme avanza el debate.
[222] 'Normal': Las tres personas
[223] 'Normal': • El Socio (GPT-4o, temp 0.9): founder agresivo, orientado a revenue. Piensa que el mayor riesgo es quedarse quieto. Usa datos: CAC, LTV, pipeline
[224] 'Normal': • El Gestor (GPT-4o, temp 0.9): lleva 20 anos viendo quebrar empresas por saltarse lo fiscal. Habla de Hacienda, AEAT, modelos 303
[225] 'Normal': • El Coach (GPT-4o, temp 0.9): su obsesion es el dueno del negocio, no el negocio. Pregunta: tiene el equipo para ejecutar eso? va a aguantar el ritmo?
[226] 'Normal': Las tres rondas
[227] 'Normal': • Ronda 1: cada persona opina sobre el dilema desde su perspectiva (40-70 palabras). Ven lo que han dicho los anteriores y pueden discrepar
[228] 'Normal': • Ronda 2: replicas: cada persona DEBE citar a un companero por nombre y decir que parte de lo que dijo es incompleta o peligrosa (30-50 palabras)
[229] 'Normal': • Ronda 3: el Secretario sintetiza el debate y genera un plan de accion con estructura fija: Veredicto, Por que, Los 3 primeros pasos, El riesgo principal
[230] 'Normal': Por que SSE y no WebSockets?
[231] 'Normal': El debate es unidireccional: el servidor envia eventos al cliente, el cliente no envia nada durante el stream. SSE es exactamente para eso y es mas simple: funciona sobre HTTP estandar, no necesita handshake especial ni gestion de reconexion compleja. WebSockets son bidireccionales y habrian sido sobreingenieria aqui.
[233] 'Normal': 8. Model Context Protocol (MCP)
[234] 'Normal': ________________________________________________________________________________
[235] 'Normal': MCP es un estandar abierto de Anthropic para exponer herramientas a agentes de IA de forma estandarizada. En vez de definir herramientas dentro del codigo del agente, se exponen a traves de un servidor MCP que cualquier agente compatible puede consumir.
[236] 'Normal': mcp_server.py / mcp_server_sse.py
[237] 'Normal': Expone las herramientas de la plataforma (get_financial_summary, get_appointments, search_web, schedule_appointment, etc.) como endpoints MCP. El servidor SSE (mcp_server_sse.py) corre en el contenedor zeptai_mcp en el puerto 8001 y permite conexion remota.
[238] 'Normal': core/mcp_client.py — Cliente MCP
[239] 'Normal': ZeptaiMCPClient soporta dos modos de transporte:
[240] 'Normal': • SSE: si MCP_SSE_URL esta definido en el entorno, conecta al servidor MCP remoto via HTTP. Usado en produccion con Docker
[241] 'Normal': • stdio: lanza mcp_server.py como subproceso y se comunica por stdin/stdout. Usado en desarrollo local
[242] 'Normal': execute_agent_loop() orquesta el agente OpenAI con las herramientas del servidor MCP: llama a GPT, recibe los tool calls, los ejecuta contra el servidor MCP, y repite hasta que GPT genera una respuesta final sin tool calls.
